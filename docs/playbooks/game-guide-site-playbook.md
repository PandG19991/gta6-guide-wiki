# 游戏攻略资讯站工作流模板

## 使用方式

本模板用于从零建设一个面向搜索流量的游戏攻略资讯站。先填写“项目简报”，再按阶段执行。每一阶段只有在质量门槛通过后才能进入下一阶段。

这是一套生产流程，不是一套固定视觉或代码主题。可复用的是决策顺序、数据约束和验收方法；游戏、受众、关键词、媒体权利、内容和品牌必须重新研究。

## 1. 项目简报

```text
游戏：
目标地区与语言：
目标用户：
核心用户任务：
商业模式：广告 / 联盟 / 会员 / 其他
发布阶段：预告期 / 预发期 / 上市期 / 长尾期
首个有效版本日期：
规范域名：
代码仓库与部署平台：
主要竞品（3-5 个）：
官方事实来源：
官方媒体来源：
明确排除：泄露、复制攻略、虚构信息、无答案占位页、其他
成功指标（90 天）：有效索引页、自然曝光、点击；如启用隐私合规分析，再定义回访、内部搜索和任务完成事件
负责人必须处理的外部事项：域名、账号验证、广告、同意管理、书面许可
```

## 2. 不变量与可替换参数

### 每个项目都保留

- 玩家任务优先于关键词数量和文章数量；
- 官方来源优先，社区内容只作为线索，泄露不进入正式页面；
- 每个索引页必须提供完整答案、相关视觉和有用下一步；
- 事实来源、媒体来源和更新时间可追踪；
- 没有答案的页面不索引，重复意图合并并永久跳转；
- SEO、安全、性能和链接要求进入自动检查；
- 上线前检查桌面、移动、键盘、图片裁切、溢出和错误状态；
- 广告在产品价值、隐私和同意条件成熟后接入。

### 每个项目重新决定

- 品牌、域名、语言、地区、配色和视觉母题；
- 搜索需求簇、页面结构、首发内容和更新频率；
- 官方与第三方来源优先级；
- 媒体权利判断、图片处理和署名方式；
- 技术栈、部署平台、分析与广告工具；
- 上市前后可验证的玩法范围。

## 3. 阶段化工作流

### 阶段 0：任务契约

**输入**：商业目标、目标市场、预算、时间、账号权限。

**动作**：

1. 明确目标、事实来源、范围与排除项、验收证据和外部操作审批边界。
2. 建立主项目状态、开发日志和独立 worktree 规则。
3. 把站长账号、验证码、购买和发布等动作列为明确交接，不把凭证写入仓库。

**输出**：一页项目简报、完成标准、风险清单。

**通过门槛**：所有人能用一句话说明“这个站帮谁完成什么任务”。

### 阶段 1：需求与竞品研究

**输入**：游戏名称、地区、语言和发布阶段。

**动作**：

1. 搜索真实用户表达，如 `walkthrough`、`where to find`、`best settings`、`map`、`missions`、`cheats`、`release date`。
2. 选择 3-5 个成熟竞品，逐项比较首页入口、攻略分类、文章模板、图片密度、更新机制、内链和广告位置。
3. 记录竞品满足了什么任务，而不是复制布局或文案。
4. 把需求分为购买决策、地图世界、资料库、即时帮助、完成度和资讯更新。
5. 为高优先级官方来源建立变化监测清单；监测结果只进入人工确认队列，不自动发布。

**输出**：搜索意图地图、竞品矩阵、首发机会列表。

**通过门槛**：每个候选页面能对应一个真实问题和一个明确结果。

**停止条件**：只有关键词、没有可提供的答案；或者只能靠泄露/猜测完成页面。

### 阶段 2：来源与媒体治理

**输入**：候选页面、官方与第三方来源。

**动作**：

1. 建立事实来源优先级：开发商/发行商、平台商店、实测、可靠媒体、社区线索。
2. 建立媒体清单，至少记录 `id`、主体、所有者、来源 URL、使用依据、状态、署名、alt、尺寸和允许位置。
3. 将媒体状态分为 `approved`、`pending`、`blocked`；渲染层只允许 approved。
4. 预发阶段优先官方嵌入、官方媒体中心、原创图解；上市后加入带版本与平台记录的第一方截图。
5. 为有争议资产设计快速替换/撤下路径。

**输出**：来源登记、媒体清单、使用规则。

**通过门槛**：代表性文章中的每个关键事实和每张图都可追溯。

**停止条件**：来源不明、图片来自搜索结果页、需要伪造 UI、或素材权利风险无法接受。

### 阶段 3：信息架构与内容生命周期

**输入**：搜索意图地图、可用事实和媒体。

**动作**：

1. 用玩家任务组织首页、分类、文章和资料库，不以“博客/证据/SEO”组织导航。
2. 给每条内容指定状态：`internal`、`draft`、`public-indexable`、`public-noindex`、`redirected`、`withdrawn`。
3. sitemap 和 RSS 只从 public-indexable 内容生成；列表和内链可包含仍对用户有价值的 public-noindex 页面，但不能把 internal、draft、redirected 或 withdrawn 内容当作正常目的页。
4. 合并同一答案的长尾页面，用章节和锚点承接相关搜索；保留 301。
5. 先画 10-20 个有用页面的内容图，不设置为了数量而设置的最低 URL 指标。

**输出**：站点地图、路由生命周期表、首发内容清单。

**通过门槛**：每个可索引路由都有独立意图、完整答案、可用视觉和下一步。

### 阶段 4：视觉原型与文章样板

**输入**：品牌方向、代表性内容、批准媒体。

**动作**：

1. 先完成首页、文章、分类/资料库三种代表页面，而不是一次铺开全站。
2. 首屏直接出现游戏/对象、搜索或主任务入口，并露出下一内容区。
3. 文章顺序默认采用：面包屑 -> 主题视觉 -> 简短答案 -> 步骤/比较/地图 -> 相关攻略 -> 简洁来源。
4. 使用稳定图片比例和尺寸，避免布局偏移；图片必须帮助识别角色、地点、道具、车辆或动作。
5. 在桌面和移动端检查真实最长标题、最小屏幕、菜单、搜索、表格和图片裁切。

**输出**：三类高保真页面、视觉规则、响应式验收截图。

**通过门槛**：不看站名也能在首屏认出这是该游戏的攻略产品；用户 5 秒内能找到一个具体任务入口。

### 阶段 5：首批内容生产

**输入**：已通过的样板、首发内容清单。

**动作**：

1. 先做高意图且可完整回答的页面，不平均填满所有分类。
2. 每页给出直接答案、具体事实/步骤/比较、相关图片、更新时间、相关下一步和必要来源。
3. 对预发信息标明“官方已确认”和“尚未确认”，但不把内部验证流程展示给用户。
4. 上市后实测记录平台、游戏版本、补丁、前置条件、方法、结果、失败情况、复现次数、剧透等级和截图。
5. 资讯页面只发布会改变玩家决策或攻略内容的更新，并回链到受影响攻略。

**输出**：首批完整攻略、资料库和更新机制。

**通过门槛**：随机抽取页面时，正文不是“未来会更新”、泛泛背景或关键词改写。

### 阶段 6：DBS 内容审计

逐页回答：

- 这是玩家要看的内容，还是内部任务清单？
- 第一屏是否回答搜索问题？
- 去掉 SEO、证据、流程和免责声明后，还有多少真实价值？
- 这个页面是否与另一页面提供同一答案？
- 图片是否展示真实对象，而不是装饰？
- 页面承诺是否与当前覆盖一致？
- 是否因为过度谨慎而把风险说明放在用户面前？

**输出**：`保留 / 重写 / 合并 / noindex / 重定向 / 撤下` 清单。

**通过门槛**：公开页面没有发布队列、验证计划、搜索词、内部 KPI、SEO 安全宣言或空洞“即将更新”。

### 阶段 7：SEO、安全与发布门槛

**构建检查**：

- 唯一 title、description、canonical 和 H1；
- 可解析且与页面可见内容一致的结构化数据；
- sitemap 只含规范、可索引、返回 200 的 URL；
- robots 指向规范 sitemap，并按 User-agent 分组正确解析；
- 内链、锚点、重定向和 404 正确；
- OG/Twitter 图片同源、存在、有尺寸和正确类型；
- 图片有来源记录、alt、响应式变体和稳定尺寸；
- RSS、manifest、security.txt 和安全头有效；
- 生产依赖没有超过项目阈值的已知漏洞。

先保留上述核心门槛。只有真实事故、重复回归或明确高风险能证明价值时，才增加新的专用检查；不要把某一个站点累积的全部断言原样复制到新项目。

**体验检查**：

- 桌面与移动无横向溢出、重叠和错误裁切；
- 搜索、菜单、筛选、键盘焦点、空状态和错误状态可用；
- 首屏 LCP 媒体受控，折叠以下图片延迟加载；
- 不在导航、按钮或交互控件附近预留误点广告位。

**通过门槛**：自动检查、代表性浏览器路径和人工截图审核全部有证据。

### 阶段 8：部署与规范域名

**动作**：

1. 部署前记录 commit 和构建结果；预览通过后再上线。
2. 只保留一个可索引的规范 HTTPS 来源；优先让 HTTP、www、平台默认域名和旧域名永久跳转并保留路径与查询参数。平台不支持时，关闭、解绑或 noindex 替代来源，结果必须是不产生可索引重复站点。
3. 在 CDN 传播窗口内使用有界重试，不因一次即时失败反复改配置。
4. 从线上重新检查 canonical、robots、sitemap、结构化数据、安全头、404、重定向和内容类型。

**输出**：部署版本、线上检查记录、回滚点。

**通过门槛**：规范 URL 返回 200，所有替代来源返回正确 301，线上质量门槛通过。

### 阶段 9：Search Console 与收录

**动作**：

1. 验证 Domain Property，提交唯一规范 sitemap。
2. 用设置 Googlebot User-Agent 的独立可抓取性探测确认状态码、类型、字节和 URL 数量；它不能证明请求来自 Google或已经被 Search Console 抓取。
3. 给首次处理留出时间，不把即时 `Could not fetch` 自动判断为站点故障。
4. 监控索引覆盖、抓取、真实查询、展示、点击和 CTR；优先改进有展示但低点击或排名接近首页的页面。

**通过门槛**：property、sitemap 记录和可抓取证据都存在。

### 阶段 10：持续增长与变现

**更新节奏**：

- 预发：只跟踪会改变玩家决策的官方节点；
- 上市周：按平台和补丁多次实测，快速更新高意图任务；
- 上市后：每周补齐地图、物品、任务、成就和版本变化；
- 长尾期：依据 Search Console 查询、内部搜索和内容缺口更新。

**增长规则**：

- 新建页面前先判断是否应更新现有页面；
- 资讯必须回流到长青攻略；
- 失效方法、补丁变化和旧版本必须有更新触发器；
- 高频共享事实应由一个数据源生成，避免价格、日期和版本在多个文件手工同步；
- 来源变化监测只创建待确认项，必须人工核对后才修改公开内容；
- 不用 AI 批量改写官方公告或竞品攻略；
- 用有效索引、搜索展示和点击衡量基础增长；只有在明确分析工具、事件、基线、隐私边界和启用阶段后，才加入回访、内部搜索、任务完成和体验事件，不用总 URL 数量衡量。

**广告门槛**：发布者身份、隐私披露、同意管理、流量基线和低布局偏移广告位全部具备后再接入。广告不能先于答案，也不能模仿导航或下载按钮。

## 4. 子代理与 worktree 协作模板

主任务负责目标、事实连续性、决策、整合和最终验收。默认采用“一名实现者 + 一次里程碑审核”；只有任务确实拥有互不依赖的工作面时才增加并行代理。下表是按需佩戴的角色帽子，不是需要同时启动的固定团队。

| 角色 | 任务 | 输出 |
| --- | --- | --- |
| 研究代理 | 搜索意图、竞品、官方事实/媒体来源 | 带链接和日期的研究表 |
| 内容审计代理 | 路由分类、重复意图、内部语言、空页面 | 页面处置清单 |
| 实现代理 | 在指定 worktree 和文件范围内完成一阶段 | 变更文件、验证命令、风险 |
| 视觉审核代理 | 桌面/移动截图、溢出、裁切、任务入口 | 可复现问题和截图 |
| 发布审核代理 | 构建、SEO、安全、重定向、线上抽查 | 通过/失败证据 |

协作规则：

1. 对会改动产品、代码或部署的阶段使用独立分支/worktree，并写明文件所有权；只读研究、文档小修或低风险单人任务不强制新建。
2. 多个代理只并行处理互不依赖的研究或文件范围。
3. 主任务不重复子代理工作，而是审查证据和整合。
4. 每阶段提交后立即记录状态，避免依赖超长对话恢复上下文。
5. heartbeat 只用于唤醒；未完成项必须来自项目状态和计划，而不是重复猜测。

## 5. 一页发布清单

```text
[ ] 页面解决一个明确玩家任务
[ ] 第一屏有直接答案或清晰任务入口
[ ] 事实、图片、更新时间可追溯
[ ] 没有泄露、复制、虚构或内部流程语言
[ ] 没有重复意图或无答案占位页
[ ] 桌面、移动、键盘和错误状态已验证
[ ] 图片尺寸、alt、裁切和加载策略正确
[ ] canonical、schema、sitemap、robots、RSS 正确
[ ] 内链、301、404 和安全响应头正确
[ ] 自动构建、内容、媒体、SEO、安全检查通过
[ ] 规范域名和平台默认域名行为已验证
[ ] Search Console 已验证并提交 sitemap
[ ] 广告、分析和同意管理符合当前阶段
[ ] commit、部署版本、验证结果和回滚点已记录
```

## 6. 何时再抽取代码模板

完成第二个不同游戏的站点后，再比较两者并抽取代码脚手架。只有同时满足以下条件才抽象：

- 两个站点都需要同一数据模型或组件；
- 差异可以通过少量配置表达，而不是大量条件分支；
- 抽取后仍能让每个游戏拥有独立视觉和内容结构；
- 自动检查能覆盖模板升级对所有站点的影响。

在此之前，复制并修改本工作流比维护一个未经验证的“万能攻略站生成器”更可靠。

## 7. 国内综合游戏门户对标校准

本节基于 2026-07-11 对游侠网和 3DM 公开页面的只读调研。两者是经营二十余年的多游戏门户，不是单游戏攻略站；可复用的是流量结构和经营指标，不是首页密度、频道数量或下载业务。

### 它们实际构建的链路

| 层级 | 游侠网与 3DM 的公开结构 | 对单游戏攻略站的校准 |
| --- | --- | --- |
| 高频发现 | 首页、滚动资讯、发售表、排行榜、促销和热点栏目 | 资讯只追踪会改变玩家决策或攻略答案的信号，不追求综合门户日更量 |
| 游戏实体中枢 | 单游戏专题/专区聚合资料、新闻、攻略、资源、评分和相关系列 | 每个重点游戏需要一个稳定 hub，承接品牌词并把用户送往具体任务 |
| 搜索长尾 | 单问题攻略、流程、收集、配置、问答和版本差异 | 保持“一页一个任务”，但所有页面必须回链 hub 和相关下一步 |
| 实用工具 | 下载、补丁、MOD、修改器、数据助手、战绩和配置工具 | 只做合法、自有且确有需求的工具，例如清单、地图、比较器或存档记录；不为了模仿门户增加下载 |
| 复访与社区 | 论坛、评论、账号、收藏、APP 和游戏社区 | 流量不足时不自建社区；先用内容更新、站内搜索和可选订阅验证复访需求 |
| 商业转化 | 品牌广告、内容推广、商城、正版促销、自运营/联运、APP 和工具服务 | 广告只是其中一层；先记录高意图出站点击，后续再验证广告、联盟或工具服务 |

公开资料能确认游侠网提供广告、资讯嵌套、活动推广、商城和联运合作；3DM 提供品牌广告、手游渠道合作、联运 SDK、自运营和商城入口。下载与补丁页面连接专区、论坛和商城，可以推断它们在高意图获客与复访中具有作用，但公开页面不能证明真实路径、“下载次数本身”就是主要收入，或各收入来源占比。

### 内容组合校准

下面的 Signal / Entity / Task / Utility 是本工作流根据对标结果提出的内部内容模型，并非两家门户公开采用的分类。下一项目的内容地图应同时检查四种库存，而不是只统计攻略页总数：

1. **Signal**：官方新闻、补丁和商店变化，目标是速度与准确性。
2. **Entity**：游戏、人物、地点、载具、任务等稳定实体页，目标是聚合与导航。
3. **Task**：回答具体问题的攻略，目标是搜索满足和下一步行动。
4. **Utility**：合法工具、清单、地图、比较器和第一方下载，目标是高意图使用与复访。

Signal 不应成为孤立资讯。每条有效资讯必须更新对应 Entity 或 Task 页面，并从资讯回链到长期页面。没有合法工具需求时，Utility 可以为空，不能用来源不明的下载、MOD 或修改器填补。

### 经营指标校准

| 阶段 | 首要指标 | 说明 |
| --- | --- | --- |
| 获客 | 各内容类型的搜索展示、点击、CTR、有效落地页 | 判断 Signal、Entity、Task 哪一层真正带来需求 |
| 分发 | hub 到攻略点击率、攻略到相关下一步点击率 | 验证网站是否形成内容网络，而不是孤立文章 |
| 更新 | 官方信号发现延迟、攻略修订延迟、过期页面比例 | 衡量速度时同时约束准确性 |
| 实用价值 | 工具完成次数、合法资源点击、已验证商店出站点击 | 下载量仅适用于自有或明确许可资源，不作为默认 KPI |
| 复访 | 回访、订阅、收藏或站内搜索复用 | 只有在隐私合规分析和真实功能存在后才启用 |
| 变现 | RPM、广告可见率、联盟转化、收入/千次高意图访问、CLS | 接入广告或联盟后才统计，不能反向驱动薄页面 |
| 内容健康 | 有需求且有答案的索引页比例、纠错率、来源覆盖、更新覆盖 | 替代“总 URL 数”和“文章总字数” |

游侠网和 3DM 页面上展示的浏览量、用户量、论坛量和日访问量只能作为其自述或页面计数，不能直接作为新站目标。单游戏新站首先需要证明 `搜索进入 -> hub/攻略分发 -> 有用下一步`，再讨论门户级 PV、社区和下载规模。

### 明确不复制

- 不复制多游戏门户的巨型导航、泛娱乐资讯和首页高密度模块。
- 不使用泄露、传闻或来源不明下载换取更新频率。
- 不把外部下载器、修改器、MOD 或软件捆绑当作默认商业路径。
- 不在没有规模和治理能力时建设论坛、账号、APP 或多子域网络。
- 不把广告位数量、翻页数量、页面浏览量或下载按钮点击当作玩家价值。
